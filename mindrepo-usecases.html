UC-01Auth

Đăng ký & Provision tài khoản

Người dùng đăng ký lần đầu qua Auth0. Hệ thống tự động tạo bản ghi PostgreSQL và sync metadata ngược lên Auth0 để các lần login tiếp theo có đủ claims.

5
Giá trị
3
Độ khó
P0
Ưu tiên
Tác nhân chính: Guest (chưa có tài khoản)
Service: user-service :8081
Phụ thuộc: Auth0, PostgreSQL, Redis
Điều kiện tiên quyết
Người dùng chưa có tài khoản. Auth0 tenant đã được cấu hình với post-user-registration action và Management API token.
Điều kiện hậu (Post-condition)
Bản ghi user_svc.users được tạo với auth0_id và email. Auth0 app_metadata chứa pg_user_id và onboarding_completed: false. Token mới được issue, Next.js redirect sang /onboarding.
Luồng chính
#Tác nhânHành độngPhản hồi hệ thống
1
GuestNhấn "Đăng ký" — chọn Google OAuth hoặc email/passwordRedirect sang Auth0 Universal Login page
2
Auth0Xác thực xong, trigger post-user-registration actionAction gọi POST /internal/provision trên user-service
3
User ServiceNhận auth0_id + email + name + pictureIdempotency check: nếu auth0_id đã tồn tại → skip. Nếu không → tạo bản ghi mới
4
User ServiceGọi Auth0 Management API: cập nhật app_metadatapg_user_id, onboarding_completed: false được ghi vào Auth0
5
Next.jsNhận callback từ Auth0 với token ban đầu (chưa có app_user_id)Gọi getAccessTokenSilently({cacheMode:'off'}) để lấy token mới có đủ claims
6
Next.js MiddlewareDecode token, phát hiện onboarding_completed: falseRedirect về /onboarding, chặn tất cả route khác
Luồng thay thế
A1 — Provision webhook thất bại
Auth0 action không gọi được user-service
Auth0 action retry tối đa 3 lần. Nếu vẫn thất bại, user login thành công nhưng token không có app_user_id. Next.js middleware phát hiện, gọi POST /auth/provision trực tiếp từ client. Đây là fallback path đảm bảo provision luôn xảy ra.
A2 — User đã tồn tại (re-provision)
Gọi provision cho user đã có bản ghi
User service kiểm tra auth0_id trước khi insert. Nếu đã tồn tại: trả về bản ghi hiện có, không tạo duplicate. Token được refresh với data hiện tại.
Yêu cầu phi chức năng
  • Provision endpoint phải idempotent — gọi N lần với cùng auth0_id cho kết quả như nhau
  • Toàn bộ flow (Auth0 → provision → token refresh) hoàn thành trong < 3 giây
  • Không lưu bất kỳ credential nào — chỉ lưu auth0_id làm foreign reference
UC-02Auth

Onboarding Profile

Người dùng mới điền thông tin bắt buộc sau lần đăng nhập đầu tiên. Hệ thống chặn tất cả route cho đến khi onboarding hoàn tất. Đây là thời điểm thu thập username duy nhất dùng trong URL repo.

5
Giá trị
2
Độ khó
P0
Ưu tiên
Tác nhân: New User (onboarding_completed: false)
Service: user-service :8081
Thông tin bắt buộc
username — 3–32 ký tự, chỉ a-z 0-9 -, unique toàn platform, dùng trong URL: mindrep.io/@username/repo-name
domains — ít nhất 1 domain quan tâm (e.g. machine-learning, physics)
Thông tin tự động điền (từ OAuth provider)
display_name lấy từ Google name, avatar_url lấy từ Google picture. User có thể thay đổi sau.
Luồng chính
#Tác nhânHành độngPhản hồi hệ thống
1
Next.jsRender trang /onboarding, prefill display_name và avatar từ GoogleForm với 2 trường bắt buộc: username + domains selector
2
UserNhập username, hệ thống validate realtimeGET /users/check-username?u=xxx — trả về available/taken ngay khi user dừng gõ (debounce 400ms)
3
UserChọn ít nhất 1 domain từ danh sách gợi ýUI highlight domain được chọn
4
UserSubmit formPATCH /users/me/onboarding — validate server-side, lưu DB trong transaction
5
User ServiceUpdate app_metadata trên Auth0: username, onboarding_completed: trueTrả về {next: "/dashboard", refresh_token: true}
6
Next.jsGọi getAccessTokenSilently({cacheMode:'off'})Token mới có onboarding_completed: true, middleware cho phép vào dashboard
Luồng thay thế
A1 — Username bị trùng race condition
Hai user chọn cùng username trong cùng lúc
Server-side dùng INSERT ... ON CONFLICT DO NOTHING kết hợp unique constraint. Nếu conflict: trả về lỗi 409 username_taken, frontend hiển thị inline error, gợi ý 3 biến thể username.
Yêu cầu phi chức năng
  • Username check debounce 400ms — không spam API khi user đang gõ
  • Update DB và Auth0 phải atomic — nếu Auth0 call thất bại thì rollback DB
  • Username regex: ^[a-z0-9][a-z0-9-]{1,30}[a-z0-9]$
UC-03Repo

Tạo & quản lý repo

Người dùng tạo repo — đơn vị tổ chức tài liệu cơ bản của MindRepo. Repo có URL dạng /@username/repo-slug, có thể public hoặc private, và quản lý danh sách contributor với phân quyền.

5
Giá trị
2
Độ khó
P0
Ưu tiên
Tác nhân: Repo Owner (authenticated user)
Service: repo-service :8082
Free tier limit: 20 repos, chỉ public
Luồng chính — Tạo repo mới
#Tác nhânHành độngPhản hồi hệ thống
1
OwnerNhấn "New Repo", điền: tên, mô tả, visibility, tags domainValidate realtime: slug availability check, tên hợp lệ
2
OwnerSubmit POST /reposTạo record trong repo_svc.repos, sinh repo_id và canonical URL
3
Repo ServiceTạo Qdrant collection cho repo: repo_{id}Collection sẵn sàng nhận embeddings
4
Repo ServiceTạo MinIO bucket prefix: repos/{repo_id}/Storage path sẵn sàng cho upload
5
SystemRedirect owner về trang repo mới tạoRepo trống, hiển thị prompt "Upload tài liệu đầu tiên"
Luồng thay thế
A1 — Thêm contributor
Owner mời người khác đóng góp vào repo
POST /repos/{id}/contributors với email hoặc username. Hệ thống gửi email mời qua Notification service. Contributor nhận vai viewer hoặc editor.
A2 — Đổi visibility
Chuyển repo từ public sang private (cần plan Pro)
Nếu user đang ở free tier: trả về 403 kèm prompt "Nâng cấp Pro để dùng private repo". Nếu đã Pro: update visibility ngay lập tức.
Yêu cầu phi chức năng
  • Slug format: ^[a-z0-9][a-z0-9-]{0,62}[a-z0-9]$, unique per user
  • Tạo repo (bao gồm Qdrant collection) hoàn thành < 2 giây
  • Xóa repo cascade: xóa tất cả documents, Qdrant collection, MinIO objects, annotations, reviews
UC-04Repo

Upload & index tài liệu

Tài liệu được upload lên MinIO qua presigned URL, sau đó Indexing Worker tự động parse, chunk, embed và lưu vào Qdrant. Đây là điều kiện tiên quyết để AI Tutor hoạt động.

5
Giá trị
3
Độ khó
P0
Ưu tiên
Tác nhân: Repo Owner / Editor
Services: repo-service indexing-worker ai-service
Formats: PDF, DOCX, MD, TXT — tối đa 50MB/file
Luồng chính
#Tác nhânHành độngPhản hồi hệ thống
1
OwnerKéo thả file vào upload zoneFrontend validate: format, kích thước. Hiển thị preview danh sách file
2
FrontendPOST /repos/{id}/documents/presign — xin presigned URLRepo service tạo presigned PUT URL cho MinIO, TTL 15 phút
3
FrontendPUT file trực tiếp lên MinIO qua presigned URLProgress bar realtime qua XHR upload events
4
Repo ServiceNhận MinIO webhook s3:ObjectCreated — tạo document recordLưu metadata vào repo_svc.documents, status: pending
5
Repo ServicePush job vào Redis queue: indexing_jobsUI cập nhật trạng thái: "Đang xử lý…"
6
Indexing WorkerConsume job: download từ MinIO → parse (Unstructured.io) → chunk → embedUpsert vectors vào Qdrant collection của repo
7
Indexing WorkerUpdate document status → indexed, publish document.indexed eventUI cập nhật: "Đã index ✓". Notification service gửi email cho contributors
Luồng thay thế
A1 — File bị mã hóa / không đọc được
Unstructured.io không parse được
Worker cập nhật status → failed với error message cụ thể. UI hiển thị "Không thể đọc file — thử xuất lại dưới dạng PDF không mã hóa". Nút "Retry" cho phép thử lại.
A2 — Worker retry logic
Lỗi tạm thời (network, OOM)
Celery retry tự động: lần 1 sau 2 phút, lần 2 sau 10 phút, lần 3 sau 30 phút. Sau 3 lần: status failed_permanent, cần can thiệp thủ công.
A3 — Batch upload
Upload nhiều file cùng lúc
Tối đa 20 file per batch. Mỗi file có presigned URL riêng. Processing song song — file nào xong trước hiển thị "indexed" trước. File lỗi không ảnh hưởng các file khác.
Yêu cầu phi chức năng
  • Indexing một PDF 100 trang hoàn thành trong < 3 phút
  • AI Tutor chỉ trả lời về document sau khi status = indexed
  • Chunking strategy mặc định: recursive text splitter, chunk size 512 tokens, overlap 64 tokens
  • Mỗi chunk lưu payload: doc_id, repo_id, page_number, chunk_index, text_preview

UC-07AI

Tương tác AI Tutor

Core differentiator của MindRepo. Mỗi repo có AI tutor riêng chỉ trả lời dựa trên nội dung tài liệu trong repo. Câu trả lời được stream token-by-token và luôn kèm trích dẫn nguồn cụ thể.

5
Giá trị
4
Độ khó
P0
Ưu tiên
Tác nhân: Learner
Service: ai-service :8090 (Python FastAPI)
Protocol: Server-Sent Events (SSE)
RAG Pipeline (chi tiết)
1. Hybrid retrieval: Dense search (text-embedding-3-small) + Sparse search (BM25/SPLADE) trên Qdrant collection của repo, fuse bằng RRF, top-20 candidates
2. Reranking: BGE-reranker cross-encoder score tất cả 20 candidates, chọn top-4
3. Context expansion: Parent-child chunking — retrieve chunk nhỏ (precision) nhưng đưa parent chunk vào context (recall)
4. Generation: Stream qua Claude API, enforce citation format trong system prompt
Luồng chính
#Tác nhânHành độngPhản hồi hệ thống
1
LearnerNhấn icon AI Tutor hoặc bôi đen đoạn văn → "Hỏi AI"Mở chat panel, prefill câu hỏi nếu có text được bôi đen
2
LearnerGửi câu hỏiHiển thị typing indicator, bắt đầu SSE stream
3
AI ServiceChạy hybrid retrieval + reranking trên QdrantLog trace vào LangFuse: retrieval latency, chunks retrieved, reranker scores
4
AI ServiceStream response từ Claude API qua SSEToken-by-token xuất hiện trong UI. Cuối response: citation list với link đến tài liệu + trang
5
AI ServiceDetect key concept → gợi ý flashcardHiển thị card preview "Lưu làm flashcard?" với Q&A auto-generated
Luồng thay thế
A1 — Câu hỏi ngoài phạm vi repo
Similarity score < 0.65 với tất cả chunks
AI phản hồi: "Tôi không tìm thấy thông tin này trong tài liệu của repo. Bạn có muốn tôi trả lời từ kiến thức chung không?" — user tự quyết định, không tự động fallback.
A2 — Session quá dài
Conversation history vượt context window
Sau 10 turns: AI tự động summarize conversation history, nén lại thành system context ngắn gọn để tiếp tục mà không mất coherence.
Yêu cầu phi chức năng
  • First token xuất hiện < 1.5 giây sau khi gửi câu hỏi
  • Citation accuracy ≥ 90%: link đến đúng đoạn văn trong tài liệu gốc
  • Không bao giờ trả lời ngoài nội dung repo khi không được user cho phép
  • Toàn bộ RAG pipeline traced qua LangFuse: latency per step, token cost, retrieval quality score

UC-11Community

Review & đánh giá repo

Reputation-weighted review system. Review từ domain expert có trọng số cao hơn — chống spam rating, tăng chất lượng curation. Phải đọc ≥ 30% nội dung mới được review.

4
Giá trị
3
Độ khó
P1
